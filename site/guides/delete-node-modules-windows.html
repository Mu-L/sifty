How to delete all node_modules folders on Windows

Old projects you have not touched in months still hold a full node_modules tree, and they add up to gigabytes. Here is how to clear every one of them across your machine, without Explorer hanging on the delete.

Works on Windows 10 and 11. Recycle Bin only, so it is reversible.

Why node_modules is such a problem

A single node_modules can hold tens of thousands of tiny files nested many levels deep. That is fine while you are working on a project, but most of us have a dozen old repos sitting around, each carrying a few hundred megabytes of dependencies we will never run again. The total is often several gigabytes hiding in folders you forgot about.

Deleting them by hand is genuinely painful on Windows. File Explorer counts every file before it deletes anything, so it can sit on "Calculating" for minutes. Worse, deeply nested dependency paths frequently exceed the old 260-character path limit, and the delete fails partway through with a "path too long" error.

The manual ways (and why they drag)

Inside one project you can usually just run a delete from PowerShell: 

PS C:\> Remove-Item -Recurse -Force .\node_modules

That works for a single folder, but it still hits the long-path limit on some packages, and it does nothing for the other eleven projects scattered across your drive. People reach for tools like npkill or a hand-written script that walks every directory. Those help, but you are now trusting a recursive force-delete to find the right folders and never touch anything else.

The one-pass way with Sifty

Sifty has a purge command built for exactly this. It walks down from where you run it, finds every known build and dependency directory, and shows you the total before touching anything: 

PS C:\> sifty purge clean
Scanning for build and dependency artifacts...
  node_modules   4.7 GB   across 9 folders
  dist / build   612 MB
  __pycache__    88 MB
Reclaimable: 5.4 GB  (dry-run, nothing deleted)

That first run is a dry-run. Nothing is deleted until you add --apply, and even then it asks before it moves anything: 

PS C:\> sifty purge clean --apply

Everything goes to the Recycle Bin, not a permanent delete, so if you cleared a project you actually needed, sifty undo brings it straight back (you would just reinstall with npm install anyway, but the safety net is there).

What purge actually targets

It only removes well-known generated directories, never your source. Out of the box that includes node_modules, .next, .nuxt, .svelte-kit, dist, build, out, target (Rust and Java), .gradle, __pycache__, .pytest_cache, .mypy_cache, .tox, bin and obj (.NET), .dart_tool, vendor and a handful of cache folders. When it matches one of these, it treats the whole directory as a single unit and does not waste time descending into it.

Tip: have a generated folder Sifty does not know about? Add it once in your config with purge.extra_patterns and it will be picked up on every future run.

A safe routine

Run sifty purge clean from the parent folder that holds your repos (something like C:\Users\you\code), read the preview, then re-run with --apply when the numbers look right. Do it once a month and those abandoned dependency trees never pile up again.
